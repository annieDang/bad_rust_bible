Commandment I: Clone Thy Neighbour

Insights into Rust Ownership and the "Dark Art" of .clone()

The Temptation of .clone()

When the borrow checker raises its eyebrow, developers often reach for .clone() as a universal silencer. It's a semantic // FIXME that often stays forever.

The Problem: Function signatures that demand ownership (T) when they only need to read data (&T) force a cascade of expensive heap allocations.

Bad Example

fn log_config(config: Config) {
    // Why own the whole struct just to print a field?
    println!("URL: {}", config.database_url.clone());
}

fn init(raw_url: String) {
    let config = Config { database_url: raw_url.clone(), ... };
    log_config(config.clone()); // Excessive cloning!
}

What it Breaks:

  • Performance: Every clone of a String or Vec triggers a new Heap allocation.
  • Correctness: Clones create "divergence points." Mutating one copy doesn't affect the other.

Noise:

Calling .clone() on Copy types (like u32) is semantically meaningless noise that confuses readers.

The 3 Laws of Ownership

From Chapter 4 of The Book. These rules are enforced at compile time — zero runtime cost.

01

Single Owner

Every value has exactly one variable that owns it at any moment.

02

One at a Time

Ownership can be transferred (moved), but only one owner can exist simultaneously.

03

Scope Drops

When the owner goes out of scope, Rust automatically frees the memory. No GC needed.

🔬 Interactive: What Happens in Memory?

Step through a simple move to see exactly what Rust is doing under the hood.

Step 1 / 5
Step 1 — Bind a String
let s1 = String::from("hello");
//  ^^
//  s1 is the OWNER

String::from allocates memory on the heap for the string data. The stack variable s1 holds a 3-part struct: a pointer, a length, and a capacity.

Memory Layout
Stack
s1 ptr | len=5 | cap=5 ↘
Heap
h e l l o
Step 2 — Move: s1 → s2
let s1 = String::from("hello");
let s2 = s1; // MOVE — s1 is gone
//  ^^
//  s2 is now the owner

Rust copies the stack struct (pointer + length + capacity) to s2, then immediately invalidates s1. The heap data is untouched. This is a move, not a copy.

After Move
Stack
s1 ptr | len | cap // ✖ invalid
s2 ptr | len=5 | cap=5 ↘
Heap
h e l l o
Step 3 — Using s1 After Move = Compile Error
let s1 = String::from("hello");
let s2 = s1;

// ❌ COMPILE ERROR:
println!("{}", s1);
// error[E0382]: borrow of moved value: `s1`
This is the borrow checker preventing a use-after-free bug at compile time. In C/C++, this would silently compile and potentially crash or corrupt data at runtime.
Why not just copy?

Rust could copy the heap data too (a deep copy), but that's expensive. Instead it moves by default and forces you to be explicit when you want an expensive copy: .clone().

Step 4 — Clone: Explicit Deep Copy
let s1 = String::from("hello");
let s2 = s1.clone(); // Deep copy

// ✅ Both are valid
println!("{} and {}", s1, s2);

.clone() explicitly allocates new heap memory and copies all data. Both s1 and s2 now own separate heap allocations. This is valid, but costs memory and time — which is why it should be deliberate.

After Clone
Stack
s1 ptr₁ | len=5 | cap=5 ↘
s2 ptr₂ | len=5 | cap=5 ↘
Heap
alloc₁ h e l l o
alloc₂ h e l l o
Step 5 — Copy Types: The Exception
let x: i32 = 42;
let y = x; // COPY — x is still valid!

// ✅ Both valid, no clone needed
println!("{} and {}", x, y);

// Compare with non-Copy:
let s1 = String::from("hello");
let s2 = s1; // MOVE — s1 gone!

Simple stack-only types like i32, bool, f64, and char implement the Copy trait. Their entire value lives on the stack, so copying them is always cheap — Rust does it automatically.

Copy types

i8..i128, u8..u128, f32, f64, bool, char, tuples of Copy types, fixed-size arrays of Copy types

Move types (not Copy)

String, Vec<T>, Box<T>, HashMap, any heap-allocated type, any struct containing a non-Copy field

📌 Borrowing Rules

Instead of moving or cloning, you can borrow a reference. Rust enforces these rules at compile time:

✓ Many Immutable Refs

Multiple readers at once is fine — data can't change, so no race conditions possible.

let s = String::from("hi");
let r1 = &s; // ok
let r2 = &s; // ok
println!("{} {}", r1, r2);
✖ Mixed Refs

A mutable ref cannot coexist with any other ref — prevents data races.

let mut s = String::from("hi");
let r1 = &s;     // ok so far
let r2 = &mut s; // ❌ ERROR!
println!("{}", r1);
✓ One Mutable Ref

A single mutable borrow is perfectly safe — exclusive access is guaranteed.

let mut s = String::from("hi");
let r1 = &mut s; // ✅ only one
r1.push_str(", world");
println!("{}", r1);
✖ Dangling Reference

Rust won't let you return a ref to a value that gets dropped. Lifetimes prevent this.

fn dangle() -> &String { // ❌ ERROR
    let s = String::from("hi");
    &s // s dropped here!
}
The Big Picture: Move semantics + borrow rules together eliminate entire classes of bugs (use-after-free, double-free, data races) that plague C and C++. The borrow checker is not your enemy — it's a proof system that your code is memory-safe.

Healing the Disease

The "disease" is demanding ownership when borrowing would suffice. Fix your function signatures to accept references.

The Righteous Refactor: Change fn f(x: String) to fn f(x: &str). 
// Accept a reference: we only need to read.
fn connect(database_url: &str) { ... }

// Borrow the whole config - no ownership needed.
fn log_config(config: &Config) {
    println!("{}", config.database_url);
}

fn init(raw_url: String) {
    let config = Config { database_url: raw_url, ... }; // Move, no clone
    connect(&config.database_url); // Borrow
    log_config(&config);           // Borrow
}

The Shared Path: Arc<T>

If you genuinely need shared ownership (e.g., across threads), don't clone the data. Clone the pointer.

use std::sync::Arc;
let shared_config = Arc::new(config);
let thread_clone = Arc::clone(&shared_config); // Cheap pointer increment

The Reviewer Smell Test

Look for these patterns during Code Review to spot "Clone-itis":

  • 🔍 Ownership Overkill: Function takes T by value but only reads fields.
  • 🔍 Hot Loop Clones: .clone() appearing inside a loop or .map() without a clear reason.
  • 🔍 Redundant Copy: Calling .clone() on types like i32, bool, or f64.
  • 🔍 Immediate Clone: let x = y.clone(); func(x); — usually means func should just take &y.
  • 🔍 The String/Vec Trap: Using &String instead of &str, or &Vec<T> instead of &[T].
Pro-Tip: Run cargo clippy. It is designed to catch redundant clones automatically!

When .clone() is Actually Fine

It is not a sin to clone if it is a deliberate architectural choice.

1. Thread Boundaries

std::thread::spawn requires 'static ownership. You often must clone data to move it into a new thread.

2. Breaking Borrow Cycles

Sometimes the borrow checker can't see that a move is safe. A clone is better than 10 lines of complex unsafe code.

3. Cheap Types

Cloning a PathBuf or a tiny String in a non-performance-critical UI component is often better for code readability.

4. Testing & Prototyping

In tests, speed of writing code matters more than nanoseconds of CPU time. Clone freely to keep tests readable.
